# Hermes 主腦分流規則 + 第一版 OpenClaw 執行任務規範

> 給新手小白看的版本。
> 目的：讓 Hermes（主腦）知道「**什麼任務該交給 OpenClaw、什麼不該**」，並用安全、可驗證的方式呼叫 `dispatch_to_openclaw`。
> 本文件只是規則與建議，**沒有修改任何程式核心**。

---

## 1. 目前角色分工

```text
使用者
  │
  ▼
Hermes（主腦）  ── 規劃 / 判斷 / 記憶 / 整理結果
  │  只有「需要執行」時才呼叫工具 dispatch_to_openclaw
  ▼
MCP tool（dispatch_to_openclaw）  ── 把工具呼叫變成 HTTP 請求
  │
  ▼
Adapter（FastAPI, port 8000）  ── 翻譯任務 + 呼叫 OpenClaw CLI
  │
  ▼
OpenClaw（執行端 / 數位員工）  ── 真正動手做事
```

| 角色 | 負責 |
|---|---|
| **Hermes（主腦）** | 任務理解、任務拆解、判斷是否需要執行端、記憶與學習、整理 OpenClaw 回傳結果 |
| **OpenClaw（執行端）** | 執行任務、自動化操作、CLI / 平台 / 工具落地，是「數位員工」的手腳 |
| **Adapter** | 把 Hermes 任務轉成 OpenClaw 可執行格式、呼叫 OpenClaw CLI、回傳結果 |
| **MCP** | 讓 Hermes 可以「用工具的方式」呼叫 Adapter（tool = `dispatch_to_openclaw`） |

**一句話**：Hermes 是大腦負責想，OpenClaw 是手腳負責做，Adapter + MCP 是中間的神經傳導。

## 2. 什麼任務「應該」交給 OpenClaw

當任務需要「**真的去做一件事**」而不是「回答一個問題」時，交給 OpenClaw：

- ✅ 需要**執行工具**（跑某個指令、操作某個程式）
- ✅ 需要**跨平台操作**（在不同系統 / 服務之間搬資料）
- ✅ 需要**自動化**（重複性流程、批次處理）
- ✅ 需要**跑 CLI**（命令列工作）
- ✅ 需要**整理成固定格式**的資料（表格、JSON、SOP、清單）
- ✅ 需要**建立檔案或流程**——但**高風險動作前要先確認**（見第 4 節分級）
- ✅ 需要執行**明確、可驗證**的工作（有清楚的「做完長這樣」標準）

判斷訣竅：如果你會對一個真人助理說「**幫我去做…**」，那就適合 OpenClaw。

---

## 3. 什麼任務「不該」交給 OpenClaw

當任務只是「想 / 聊 / 解釋」，或風險太高、授權不明時，**Hermes 自己處理或先問使用者**：

- ❌ 純聊天、閒聊
- ❌ 純解釋、純概念教學
- ❌ 純問答（不需要外部執行就能回答）
- ❌ 簡單知識性問題（Hermes 直接答更快）
- ❌ **高風險刪除資料**（除非使用者明確確認）
- ❌ **未確認的登入、付款、下單、發送訊息**
- ❌ 含**密碼 / token / cookie / API key / 個資**的任務
- ❌ 使用者**沒有明確授權**的操作

判斷訣竅：如果你會對一個真人助理說「**這個我來想就好**」或「**這要先問老闆**」，那就不要直接丟給 OpenClaw。

## 4. 任務安全分級（Safety Level）

| 等級 | 名稱 | 說明 | 處理方式 |
|---|---|---|---|
| **Level 0** | 純文字整理 | 摘要、改寫、整理格式，無副作用 | ✅ 可自動執行 |
| **Level 1** | 讀取資料 | 讀檔 / 查資料但**不修改** | ✅ 可自動執行 |
| **Level 2** | 建立草稿 | 產生草稿 / 建議檔，但**先回報**讓使用者看 | ⚠️ 自動執行 + 回報 |
| **Level 3** | 修改檔案 | 改既有檔案 / 設定 | ⛔ **需使用者確認**後才做 |
| **Level 4** | 高影響動作 | 登入、付款、刪除、下單、發送訊息 | ⛔ **必須先問使用者**，逐項確認 |
| **Level 5** | 高風險 / 不明確 | 風險高、目標不清、授權不明 | 🚫 **拒絕或請使用者補充**，不要硬做 |

> 規則：**等級越高，越要先問人。** Level 0–1 可自動；Level 2 要回報；Level 3 以上一定要使用者點頭。
> 安全等級放在 `metadata.safety_level`（見第 6 節）。

## 5. Hermes 呼叫 OpenClaw 前的自我檢查清單

每次要呼叫 `dispatch_to_openclaw` 前，Hermes 先在心裡跑一遍：

- [ ] **這件事真的需要 OpenClaw 嗎？**（能自己答就別呼叫）
- [ ] **任務目標是否明確？**（一句話講得清「做完長怎樣」）
- [ ] **輸入是否完整？**（缺資料就先問使用者）
- [ ] **是否有安全風險？**（對照第 4 節分級）
- [ ] **是否需要使用者確認？**（Level 3 以上 = 要）
- [ ] **是否可以驗證完成？**（怎麼知道做對了）
- [ ] **預期輸出是什麼？**（文字？表格？JSON？）
- [ ] **是否會消耗模型額度？**（重任務先想值不值得）
- [ ] **是否可能修改檔案或操作平台？**（會的話降速、先確認）

> 只要有任何一項是「不確定」或「高風險」，**先問使用者，不要先做**。

## 6. 標準 Task Envelope（任務信封）

Hermes 呼叫工具時，建議組成這個標準 JSON：

```json
{
  "title": "商品資料整理",
  "goal": "把雜亂商品資訊整理成統一表格",
  "task_text": "請把以下商品資訊整理成表格：名稱、價格、幣別、來源、備註。不要操作檔案、不要登入、不要下單。\n\n<這裡放原始商品資訊>",
  "priority": "low",
  "metadata": {
    "source": "hermes",
    "workflow": "product_data_cleanup",
    "safety_level": "level_0",
    "requires_confirmation": false,
    "expected_output": "Markdown 表格，欄位：名稱/價格/幣別/來源/備註"
  }
}
```

### ⚠️ 欄位重點（以 Adapter 程式實際支援為準）

- **必填欄位是 `task_text`**（完整指令）。
- **沒有 `instruction` 這個欄位**——不要用 `instruction`，會缺 `task_text` 而收到 HTTP 422。
- 目前 Adapter 真正讀的欄位：`title`、`goal`、`task_text`、`priority`（`low`/`normal`/`high`）、`metadata`。
- `metadata` 是**自由欄位**，所以 `safety_level` / `requires_confirmation` / `expected_output` **現在就能放進去傳**（會被原樣記錄與帶入），未來再考慮升級成正式欄位。
- `metadata.safety_level` / `requires_confirmation` 目前是「**給 Hermes 自己遵守的約定**」，Adapter 還不會主動依它擋下任務——所以**把關責任在 Hermes**。

## 7. Hermes 系統提示詞草稿（可放進 profile / system prompt / skill）

```text
你是 Hermes，是整個系統的主腦。你有一個工具：dispatch_to_openclaw，
它會把任務送到本機 Adapter，再由真實 OpenClaw 執行，並回傳結果。

【核心原則】
- 不要把所有任務都丟給 OpenClaw。能自己回答的（聊天、解釋、知識、簡單問答）就直接回答。
- 只有當任務需要「執行 / 工具 / 自動化 / 跨平台 / 明確落地」時，才呼叫 dispatch_to_openclaw。

【呼叫前自我檢查】
- 確認真的需要執行端、目標明確、輸入完整、可驗證完成、知道預期輸出。
- 對照安全分級：Level 0–1 可自動；Level 2 執行後要回報；Level 3 以上（改檔/登入/付款/刪除/下單/發訊息）一定先問使用者確認；Level 5（高風險或不明確）拒絕或請使用者補充。

【呼叫方式】
- 組成清楚的 task_text（完整、具體、可驗證的指令），必填。
- 在 task_text 內明確寫出安全邊界，例如「不要操作檔案、不要登入、不要下單、不要發送訊息」。
- 在 metadata 帶 source / workflow / safety_level / requires_confirmation / expected_output。
- 不要把密碼、API key、token、cookie、個資放進 task_text 或 metadata。

【收到結果後】
- 讀取工具回傳（openclaw_response），確認 ok 是否為 true、結果是否合理。
- 把結果整理成使用者看得懂的人話再回覆，不要直接貼原始 JSON。
- 若失敗，說明原因（error / 缺欄位 / 逾時 / Adapter 沒啟動）並給下一步建議。
```

## 8. 第一個實用工作流建議（3 個候選）

### A. 商品資料整理工作流
- **目標**：把雜亂的商品資訊整理成統一表格。
- **Hermes 做什麼**：收使用者貼上的原始資料、判斷這是整理任務（Level 0）、組 task_text、呼叫工具、把結果整理回報。
- **OpenClaw 做什麼**：把文字整理成指定欄位的表格。
- **輸入格式**：一段純文字（商品名稱、價格、來源等混雜）。
- **輸出格式**：Markdown 表格（名稱 / 價格 / 幣別 / 來源 / 備註）。
- **安全等級**：**Level 0**（純文字整理，無副作用）。
- **適合當第一個嗎**：✅ 非常適合（零風險、好驗證）。

### B. 代購訂單整理工作流
- **目標**：把多筆代購需求整理成標準訂單清單。
- **Hermes 做什麼**：收訂單訊息、拆出每筆品項、組 task_text、呼叫工具、整理回報、（金額/下單部分）保留給使用者確認。
- **OpenClaw 做什麼**：把雜亂訂單文字整理成結構化清單（品項 / 數量 / 單價 / 小計 / 客人 / 備註）。
- **輸入格式**：多行訂單文字 / 對話節錄。
- **輸出格式**：表格或 JSON 訂單清單 + 總計。
- **安全等級**：**Level 0–1**（只整理就好）；**一旦牽涉真的下單 / 付款 / 發訊息給客人 → 升 Level 4，必須先問使用者**。
- **適合當第一個嗎**：🔸 整理部分適合；但要嚴格限制「只整理、不下單、不發訊息」，否則容易踩到高風險。

### C. Beyblade 採購分析工作流
- **目標**：彙整某些 Beyblade 商品的資訊，做採購比較與建議。
- **Hermes 做什麼**：收商品清單 / 連結文字、組 task_text（要求只做分析整理）、呼叫工具、把比較結果整理成建議。
- **OpenClaw 做什麼**：把多筆商品資訊整理成比較表（型號 / 價格 / 來源 / 庫存註記 / CP 值備註），並給文字建議。
- **輸入格式**：一段商品資訊文字（**先不要叫它上網抓**，由使用者貼資料最安全）。
- **輸出格式**：比較表 + 三點採購建議。
- **安全等級**：**Level 0–1**（只分析既有文字）；若要它**自動上網爬 / 登入賣場 / 下單 → 升 Level 4**。
- **適合當第一個嗎**：🔸 適合，但**第一版限制成「只分析使用者貼上的資料」**，不要碰爬蟲 / 登入 / 下單。

---

## 9. 推薦第一個正式工作流

**推薦 → A. 商品資料整理工作流。**

理由（完全符合你的偏好：不登入、不下單、不改檔案、不發訊息）：
- 純文字 / 表格整理，安全等級 **Level 0**，零副作用、最好驗證。
- 輸入只要使用者貼一段文字，輸出一張表格，成功與否一眼可判。
- 跟你之後要做的「代購 / Beyblade 採購」是同一條技能線（資料整理），先把這條練穩，再往 B、C 擴張。

> 進階順序建議：**A（商品資料整理）→ C（Beyblade 分析，限定只分析貼上的資料）→ B（代購訂單整理，仍只整理不下單）**。
> 三者第一版都嚴守「只整理 / 只分析、不執行高風險動作」。
