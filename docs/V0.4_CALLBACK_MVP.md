# v0.4 — Callback MVP（非同步背景執行 + 結果回報）

> 把 Adapter 從「同步等 OpenClaw 做完」改成「**立刻回 accepted，背景執行，完成後寫結果 / callback**」。
> 不做 Queue、不做 DLQ、不改 Hermes / OpenClaw 核心，只改 Adapter 與 MCP bridge。

---

## 1. 新架構圖

```text
Hermes
  │  MCP tool: dispatch_to_openclaw
  ▼
Adapter  POST /tasks/dispatch
  │  1) 產生 task_id、寫 tasks.jsonl (queued)
  │  2) 立刻回 {"status":"accepted","task_id":...}   ← Hermes 不用等
  │  3) 背景啟動 Background Runner
  ▼
Background Runner  run_openclaw_and_callback()
  │  寫 tasks.jsonl (running) → 呼叫 OpenClaw CLI（背景）
  ▼
openclaw agent --message ... --json
  ▼
真實 OpenClaw → 回傳結果
  │  寫 data/results.jsonl（completed / failed）
  │  寫 tasks.jsonl（completed / failed）
  ▼
Callback Sender  send_callback_to_hermes()
  ├─ ledger_only（預設）：不送 HTTP，results.jsonl 就是帳本
  └─ http：POST HERMES_CALLBACK_URL（帶 HMAC 簽章）
        成功 → data/callbacks.jsonl
        最終失敗 → data/callback_errors.jsonl

之後 Hermes 用 MCP tool get_openclaw_task_result(task_id)
  → Adapter GET /tasks/{task_id}/result → 拿到 result
```

## 2. 啟動方式

```bash
cd ~/projects/hermes-openclaw-adapter
source .venv/bin/activate
uvicorn app.main:app --host 0.0.0.0 --port 8000 --env-file .env
```

健康檢查（會顯示 callback 模式）：
```bash
curl http://127.0.0.1:8000/health
# {"ok":true,...,"callback_enabled":true,"callback_mode":"ledger_only",...}
```

## 5. 如何看 tasks.jsonl（任務狀態帳本）

每個任務狀態變化會 append 一行（queued → running → completed/failed，或 rejected）：
```bash
cd ~/projects/hermes-openclaw-adapter
tail -n 5 data/tasks.jsonl | python3 -m json.tool 2>/dev/null || tail -n 5 data/tasks.jsonl
# 查某個 task 的所有狀態：
grep task-xxxx data/tasks.jsonl
```
「最新狀態」= 該 task_id 的最後一行。

## 6. 如何看 results.jsonl（結果帳本）

每個任務完成（或失敗）寫一筆 TaskResult：
```bash
tail -n 1 data/results.jsonl
```
TaskResult 結構（schema_version v1）：
```json
{
  "schema_version": "v1",
  "task_id": "task-xxxx",
  "correlation_id": "corr-xxxx",
  "status": "completed",         // 或 failed
  "finished_at": 1718350000,
  "title": "...", "goal": "...",
  "summary": "OpenClaw 已完成任務。",
  "result_text": "PONG",
  "error": null                  // 失敗時 {code,message,retryable}
}
```

## 7. 如何看 callbacks.jsonl（只有 http 模式才會出現）

```bash
tail -n 5 data/callbacks.jsonl          # 成功送出的 callback
tail -n 5 data/callback_errors.jsonl    # 重試 N 次仍失敗的 callback
```
- `callbacks.jsonl`：`{task_id, correlation_id, ts, url, status_code, attempt}`
- `callback_errors.jsonl`：`{task_id, correlation_id, ts, url, attempts, error}`

## 8. 如何切換 ledger_only / http callback

改 `.env` 的 `HERMES_CALLBACK_MODE`，然後**重啟 Adapter**：

- `ledger_only`（預設）：完成只寫 `results.jsonl`，**不送任何 HTTP**。最安全，適合先用。
- `http`：完成後額外 `POST HERMES_CALLBACK_URL`，帶：
  - `Content-Type: application/json`
  - `X-Hermes-OpenClaw-Signature: sha256=<hmac>`（用 `HERMES_CALLBACK_SECRET` 對 **payload bytes** 做 HMAC-SHA256）
  - 逾時 `CALLBACK_TIMEOUT_SECONDS`（預設 20），重試 `CALLBACK_MAX_RETRIES`（預設 3）。

接收端驗證簽章（範例）：
```python
import hmac, hashlib
expected = "sha256=" + hmac.new(SECRET.encode(), raw_body_bytes, hashlib.sha256).hexdigest()
assert hmac.compare_digest(expected, request.headers["X-Hermes-OpenClaw-Signature"])
```

## 9. 安全規則（Safety Level）

| safety_level | 行為 |
|---|---|
| 未填 | 視為 **Level 0**，但在 ledger 標 `assumed_safety_level: true` |
| Level 0 / 1 | ✅ 自動背景執行 |
| Level 2 | ⛔ 不自動執行（需人工 review）→ 回 `rejected` |
| Level 3 / 4 / 5 | 🚫 回 `rejected`：`High risk task requires human confirmation before OpenClaw execution.` |

被擋時回：
```json
{"status":"rejected","task_id":"task-xxxx","reason":"..."}
```

## 10. MCP 工具（mcp/openclaw_mcp.py）

- `dispatch_to_openclaw(title, goal, task_text, priority, metadata)` → 立刻回 `{"status":"accepted","task_id":...}`（或 `rejected`）。
- `get_openclaw_task_result(task_id)` → 回任務 `result`（未完成時 `result: null` + `status`）。

Hermes 流程：**派工 → 之後查結果**（未來可再接真正 webhook callback）。

## 11. 目前限制（v0.4 明確不做）

- ❌ 沒有 **Queue**（多任務直接各自背景跑，無排隊/限流）。
- ❌ 沒有 **DLQ**（死信佇列）；callback 最終失敗只寫 `callback_errors.jsonl`。
- ❌ 沒有 **OpenClaw 原生 webhook**；callback 是**由 Adapter 代發**。
- ❌ 沒有重試「任務本身」（只重試 callback HTTP）。
- ❌ 不持久化 process 狀態：Adapter 重啟後，正在背景跑的任務會中斷（結果不會補寫）。

> 這些都留給 **v0.5 queue-worker** 再做（見專案根 README / 下一步規劃）。
