# Phase 8 Remote Blackboard Read-only Plan

Status: **PLANNING ONLY, NOT AUTHORIZED**

This document is a design plan, not implementation authority. It defines a narrow,
read-only mobile view for the existing N=1 workflow. It does not authorize a remote
API, webhook, callback receiver, queue consumer, Blackboard writer, approval writer,
execution hook, dispatch hook, or runtime connection.

This document contains no example code. If a later planning revision adds any example
code, every example must be immediately preceded by the exact line
`PLANNING ONLY, NOT AUTHORIZED`.

## 1. Scope and invariants

The planned data direction is one way only:

**local source -> GitHub private repository -> Replit read-only deployment -> authenticated mobile browser**

The Replit side pulls from GitHub. It never initiates a connection to the local system.
There is no local receiver, remote receiver, webhook, callback, reverse tunnel, or
bidirectional synchronization. GitHub is the only planned transfer boundary.

The following invariants are mandatory design gates:

1. Local remains the source of truth.
2. Only an allowlisted, pre-sanitized display projection may cross the GitHub boundary.
3. Raw Blackboard messages and raw approval packets never cross that boundary.
4. Replit and the browser have no credential capable of writing to GitHub or local.
5. The remote application exposes GET-only authenticated pages and static assets.
6. A displayed approval state is descriptive data only. It cannot dispatch, execute,
   enqueue, write an audit event, or advance a task.
7. `single_use_execution_token` is never projected, displayed, transmitted, or changed.
8. A sync or display failure fails closed: the last verified projection is marked stale,
   or no data is shown.
9. This phase adds no remote API, webhook, receiver, POST route, queue, or runtime wiring.

## 2. N=1 data product

The only planned remote data product is one immutable display projection for one
single-worker rehearsal chain. The projection is not a Blackboard schema and is not a
transport envelope. It is a deliberately lossy view made for mobile status inspection.

The proposed allowlist is:

| Display field | Purpose | Disclosure rule |
|---|---|---|
| `projection_schema_version` | Reject unknown display layouts | Fixed version; no runtime capability implied |
| `data_generated_at` | Show when local produced the projection | UTC timestamp only |
| `source_commit_sha` | Bind the view to an immutable repository state | Full or shortened Git commit SHA; no filesystem path |
| `pulled_at` | Show when Replit obtained the commit | UTC timestamp generated by deployment/pull layer |
| `task_display_id` | Let the Owner correlate the N=1 rehearsal | Pseudonymous or truncated ID; not the raw task body |
| `parent_task_display_id` | Show the single parent relation | Pseudonymous or truncated ID |
| `phase` | Explain where the rehearsal stopped | Closed display enum; no command text |
| `status` | Show pending/ready/decided/failed display state | Closed display enum; cannot trigger transitions |
| `execution_class` | Show the already-recorded safety class | Display only |
| `safety_summary` | Surface selected Boolean safety facts | Explicit Boolean allowlist; no arbitrary keys |
| `approval_readiness` | Show whether the packet is ready for Owner review | Summary only; not a decision control |
| `decision_summary` | Show an already-recorded decision, if any | Verb and timestamp only; no execution token |
| `evidence_bundle_hash` | Allow integrity correlation | SHA-256 digest only; no evidence payload |
| `stale_after` | Define when the UI must display stale state | UTC timestamp only |

The projection must reject additional properties. It must be produced locally from
already-validated contract messages, then sanitized and schema-validated before it is
eligible for publication. The future implementation design must include deterministic
serialization and an integrity digest, but that implementation is outside Phase 8.

### Data that must never enter the projection

- raw prompts, task descriptions, annotations, comments, result bodies, or approval
  packet bodies;
- command text, command arguments, command envelopes, tool inputs, tool outputs, or
  diff contents;
- connector payloads, customer data, document contents, spreadsheet values, or account
  identifiers;
- tokens, cookies, authorization headers, credentials, private keys, environment
  variables, secrets, or `single_use_execution_token`;
- local or remote filesystem paths, hostnames, internal URLs, callback URLs, or network
  topology details;
- stack traces, request/response bodies, full logs, queue contents, or database rows;
- any unrecognized field added by a future Blackboard contract.

If a useful mobile screen would require any excluded data, Phase 8 must stop at HOLD and
return the proposed disclosure to the Owner. Convenience is not grounds to widen the
allowlist.

## 3. Planned component boundaries

Every component below has an explicit read set, prohibited write set, authentication
boundary, and leak surface. These are design requirements, not claims that the
components have been implemented.

| Component | Reads | Must never write | Authentication | Principal leak surface and control |
|---|---|---|---|---|
| Local projection producer | Validated N=1 contract records needed for the allowlist | Blackboard, queue, audit trail, approval decision, task state, runtime, or external system | Local OS identity and repository access controlled by Owner | Raw source data entering projection; prevent with explicit field projection, closed schema, secret/path scan, and fail-closed validation |
| Local publication step | Validated projection artifact and target Git branch state | Any runtime state, Replit state, or unrelated repository content | A local-only GitHub write credential scoped to the one private repository and protected branch workflow | Publisher credential or unintended Git content; keep credential out of files/logs and stage an exact file allowlist |
| GitHub private repository | Immutable commits containing application code and sanitized projection | Local source, Blackboard, queue, approval store, or runtime | Repository access controls; local publisher has minimum write scope, Replit has read-only scope | Repository collaborator access and commit history; private repository, least privilege, review, and revocation procedure |
| Replit pull/deploy layer | A pinned GitHub commit and sanitized projection | GitHub, local, Blackboard, approval store, queue, runtime, or external connector | Read-only deploy key or read-only installation credential stored only in Replit secret storage | Read credential and build logs; prohibit token-in-URL, redact logs, pin host key, rotate/revoke credential |
| Replit dashboard process | Sanitized projection from its deployed read-only checkout | Projection, GitHub, local, server-side session decisions, queue, audit trail, task state, or runtime | Existing authenticated dashboard boundary, with production-grade session configuration required before authorization | Template/debug output and cache; strict allowlist rendering, generic errors, no raw objects, no debug mode, no payload logging |
| Mobile browser | Authenticated rendered HTML and static assets | Any server, GitHub, local source, task, decision, approval, or execution state | Owner authentication over HTTPS; short-lived secure session | Browser history, screenshots, shoulder surfing, shared devices, and cached pages; no payload, cache-control, logout, re-authentication, and stale banner |
| Monitoring view | Health, commit SHA, projection timestamps, and generic error codes | Application data, decision data, payloads, or remote actions | Separate least-privilege observability access if authorized | Logs becoming a shadow payload store; structured metadata-only logging and retention limits |

The local publication step is the only planned writer in the transfer chain, and it
writes only the sanitized artifact and ordinary Git metadata to GitHub under a separate,
future authorization. It is not part of an agent loop and cannot be invoked by a packet
or dashboard state.

## 4. Authentication and credential plan

### Local to GitHub

- The private repository is the only destination.
- The publication identity must be separate from the application runtime identity.
- Its credential must remain on the trusted local side, use the minimum repository
  scope, and never be stored in the repository, projection, shell history, or logs.
- Branch protection or an equivalent reviewed publication policy should prevent an
  automated force-push and make each published state attributable.
- Credential revocation must stop new publication without affecting the local source.

### GitHub to Replit

- Replit receives a credential that is demonstrably read-only for the one repository.
- The credential belongs in Replit secret storage, never an environment display page,
  source file, Git remote URL, HTML response, or diagnostic log.
- The pull must select an immutable commit SHA. Moving branch names may be used only to
  discover a candidate; the displayed projection records the resolved SHA.
- There is no GitHub webhook. A manual or platform-native scheduled pull may be
  evaluated later, but no scheduler or receiver is authorized by this document.
- Rotation and emergency revocation must be rehearsed before production authorization.

### Replit to mobile browser

- The dashboard remains behind authentication; anonymous access fails closed.
- HTTPS, secure and HTTP-only session cookies, CSRF-safe logout semantics, expiry,
  rate limiting, and re-authentication policy are authorization gates.
- Authentication errors reveal no task identifiers, projection contents, repository
  details, or difference between existing and nonexistent tasks.
- The browser receives no GitHub or local credential.

## 5. Read-only dashboard behavior

The remote mobile view is intentionally smaller than the local dashboard. It shows a
single compact status card, integrity/staleness metadata, safety summary, and an
already-recorded decision summary. It does not display or embed review controls.

The planned remote route inventory must satisfy all of these checks before approval:

- only GET and HEAD are present for application pages and required static assets;
- no form, button, editable control, action URL, JavaScript mutation request, WebSocket,
  service-worker background sync, or hidden POST exists;
- no route accepts a packet, decision, command, upload, callback, or task identifier
  that could change state;
- templates render individual allowlisted fields rather than serializing a source object;
- error pages, source maps, debug pages, response headers, and logs contain no payload;
- the UI clearly labels the source commit, generation time, pull time, and stale state;
- an `approve` value, if displayed, is plain text and has no link or event handler.

The existing local `/dashboard/reviews` decision entry remains outside the remote view.
This plan does not duplicate it and does not create another approval surface.

## 6. Freshness, integrity, failure, and rollback

The UI must distinguish three times: local projection generation, Git commit identity,
and Replit pull. A locally generated `stale_after` value controls the visible stale
banner. Replit clock time must not silently redefine data freshness.

Before display, a future authorized pull process must verify:

1. the expected private repository and pinned commit;
2. projection schema version and closed field inventory;
3. deterministic content digest;
4. absence of prohibited fields and secret/path patterns;
5. monotonic or explicitly explained projection generation time.

Verification failure produces no new view. The deployment may retain the last verified
projection only if it is prominently marked stale; otherwise it shows a generic
unavailable page. It must not fall back to fetching local, accepting an upload, or
showing raw files.

Rollback means selecting a previously verified immutable Git commit for display. It
does not roll back a task, approval decision, Blackboard entry, queue, or execution.
Rollback authority and audit procedure must be specified before implementation.

## 7. Leak-surface review

The minimum review before authorization includes:

| Surface | Failure to test | Required planning control |
|---|---|---|
| Git history | Deleted payload remains in an old commit | Never commit payload; scan the exact staged artifact before first publication |
| Build/deploy logs | Credential, remote URL, path, or projection body is printed | Metadata-only logs, token redaction, no shell tracing, retention limit |
| HTML and assets | Raw object is embedded in markup, source map, or JavaScript | Server-render allowlisted scalars; disable source maps containing data |
| HTTP cache | Authenticated page persists on a shared device or intermediary | Private/no-store caching policy and logout/re-authentication test |
| Error handling | Traceback exposes environment, paths, or data | Generic client errors; protected server diagnostics with no payload |
| Repository access | Replit credential can write or enumerate unrelated repositories | One-repository read-only credential and an explicit failed-write test |
| Staleness | Old safe-looking state is mistaken for current truth | Generation/pull timestamps, `stale_after`, obvious stale banner |
| Mobile device | Screenshot or notification preview exposes sensitive context | Display only pseudonymous summary data; no sensitive notification body |

## 8. OWASP Top 10 for Agentic Applications 2026 cross-check

The official source was available, so the checklist is included rather than skipped.
Source names were checked against the OWASP GenAI Security Project's official
[resource page](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/)
and [official publication](https://genai.owasp.org/download/52117/?tmstv=1765059207)
on 2026-07-19. This is a Phase 8 planning crosswalk, not a compliance claim or a
substitute for reading the OWASP guidance.

| OWASP item | Phase 8 read-only planning check | Remaining authorization evidence |
|---|---|---|
| ASI01: Agent Goal Hijack | Remote content cannot supply instructions to an agent; the projection is data-only and no runtime is connected | Prove no displayed field is later consumed as an instruction |
| ASI02: Tool Misuse and Exploitation | The remote process receives no execution, dispatch, connector, queue, or write tool | Route/tool inventory and negative capability tests |
| ASI03: Identity and Privilege Abuse | Separate local publisher, read-only Replit pull identity, and authenticated Owner browser | Credential-scope evidence, rotation, revocation, and failed-write test |
| ASI04: Agentic Supply Chain Vulnerabilities | Pull a reviewed private repository commit by immutable SHA | Dependency pinning, provenance, review, and deployment integrity policy |
| ASI05: Unexpected Code Execution (RCE) | Projection contains no command, arguments, code, template, or raw payload; dashboard has no execution path | Parser/template fuzzing, dependency scanning, and hardened runtime evidence |
| ASI06: Memory & Context Poisoning | Remote view has no agent memory and cannot write back; projection is schema-closed and integrity-bound | Demonstrate untrusted source text cannot enter the projection |
| ASI07: Insecure Inter-Agent Communication | No agent-to-agent channel, webhook, callback, remote receiver, or local reverse connection exists | Dynamic endpoint/network inventory showing Git pull is the sole ingress |
| ASI08: Cascading Failures | Sync failure fails closed or shows a stale last-known-safe projection; it cannot advance workflow state | Stale, corrupt, missing, rollback, and partial-deploy rehearsals |
| ASI09: Human-Agent Trust Exploitation | UI labels data time, source SHA, staleness, and read-only status; `approve` is inert text | Mobile usability test showing state cannot be mistaken for an action prompt |
| ASI10: Rogue Agents | No remote agent is deployed and no remote component has Blackboard or runtime credentials | Deployment inventory and continuous assertion that agent/runtime imports are absent |

## 9. Authorization gates and HOLD conditions

Implementation remains unauthorized until the Owner separately approves all of the
following:

1. the final closed projection schema and every display field;
2. repository privacy, collaborator list, retention, and incident response;
3. the exact local publication mechanism and its write credential;
4. the exact GitHub-to-Replit read-only authentication mechanism;
5. mobile authentication, session, cache, and logout behavior;
6. dynamic proof that the remote app has no write route, tool, queue, connector,
   runtime, execution, dispatch, or packet-consumption path;
7. secret/path/payload scans and corrupt/stale/rollback rehearsals;
8. credential rotation and emergency shutdown procedure.

The work must enter HOLD if any design requires a new endpoint, webhook, callback,
receiver, upload, remote write, bidirectional sync, packet consumer, approval control,
execution hook, dispatch hook, queue connection, runtime import, or disclosure outside
the approved projection allowlist.

## 10. Explicit non-deliverables

Phase 8 does not deliver or authorize:

- remote API, webhook, callback, receiver, tunnel, sync daemon, or scheduler;
- Replit application code, deployment configuration, credential, or repository setup;
- projection builder, serializer, schema, validator, publisher, or Git automation;
- dashboard route, template, control, login change, POST action, or JavaScript client;
- approval decision handling, token issuance, task mutation, queueing, execution,
  dispatch, connector access, or runtime integration.

Those items require a later, separately scoped package. Any such package must preserve
the one-way GitHub-to-Replit pull boundary and prove that remote data remains a lossy,
read-only display projection.
