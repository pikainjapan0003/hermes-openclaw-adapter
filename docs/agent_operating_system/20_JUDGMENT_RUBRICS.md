# 20 判斷力 Rubrics（Judgment Externalized）

- 讀者：弱模型。目的：把「該不該、算不算、停不停」這類判斷變成可機械執行的檢查。
- 格式：每條 = 判準／觸發條件／操作／PASS 條件／HOLD 條件／正例／反例。

---

## R-01 何時該升級模型

- 觸發條件：任一成立——(a) 同一子任務已失敗一次且你是最便宜等級；(b) 連錯兩次；(c) 任務含「取捨」「設計」「該不該」字眼；(d) 涉及安全邊界的新代碼。
- 操作：按 10_MODEL_ORCHESTRATION.md C5 升級，附完整失敗軌跡。
- PASS：升級請求附了原 prompt + 失敗輸出 + 期望。
- HOLD：無更強模型可用 → 回報 Owner「已到本環境最強模型，仍失敗兩次」。
- 正例：schema validator 測試紅兩次 → 停止重試，打包軌跡升級。
- 反例：「我再試一次說不定就好了」×5。

## R-02 何時算真的完成

- 判準：完成 = 產物落檔 + 客觀驗收通過 + 驗收者 context 乾淨（或已標註自驗）。
- 觸發條件：你想說「完成了」的每一刻。
- 操作：自查三項：檔案在磁碟上嗎（不是在回覆草稿裡）？驗收是客觀信號（測試/read-back/diff）還是「我覺得」？驗收者是誰？
- PASS：三項皆有具體證據可引用。
- HOLD：任何一項答不出 → 不得宣稱完成，改說「產出已落檔，驗收待做」。
- 正例：「05 已寫入，fresh subagent read-back 確認 12 章節齊全。」
- 反例：「計劃表寫好了，內容我很有信心。」（沒落檔、沒驗收）

## R-03 何時該停下來問 Owner

- 觸發條件：(a) 動作不可逆且授權引用不出逐字句；(b) 兩份權威文件衝突且無優先序；(c) 任務是品味/方向/商業取捨；(d) 重試已達兩輪；(e) 發現三源不一致。
- 操作：停該項，用一句話 + 一個具體問題問 Owner；同指令內其他已授權項繼續做。
- PASS：問題具體到 Owner 能用一句話回答。
- HOLD 本身即是操作。
- 正例：「data/audit_dev.jsonl 已存在且含未知格式資料。覆蓋還是另開新檔？」
- 反例：「請問我該怎麼做比較好？」（沒有選項、沒有背景）；或反過來：怕打擾 Owner 而自行刪掉了未知檔案。

## R-04 什麼訊號代表方向錯了（換路而非重試）

- 觸發條件：(a) 每次修好一個錯就冒出一個新錯，連續三次；(b) 為通過驗收而不斷放寬驗收；(c) workaround 疊 workaround；(d) 你開始想改測試而不是改代碼。
- 操作：停止當前路線。寫下「目前路線、為何不通、備選路線」三行，換路或升級。
- PASS：換路決定附了三行分析。
- HOLD：沒有備選路線 → 問 Owner。
- 正例：「UNC 路徑下 git 太慢改不動 → 換路：git 全部進 wsl.exe 執行。」
- 反例：第四次調整同一個正則想讓它同時滿足五個 case。

## R-05 品質底線怎麼驗

- 判準：每類產物的最低驗收——文件：必要章節齊 + 引用路徑存在 + read-back；代碼：測試綠 + lint 過 + 至少一個反例測試；計劃：每階段有驗收與 HOLD 條件；命令建議：實跑過或標註「未實跑」。
- 觸發條件：交付任何產物前。
- 操作：對照上行清單逐項打勾。
- PASS：全勾。HOLD：缺項且補不了 → 交付時明列缺什麼。
- 正例：「代碼交付：pytest 7 綠、含 2 反例、ruff 過。」
- 反例：「寫完了，應該沒問題。」

## R-06 何時應該 HOLD（授權判定）

- 判準：授權只存在於 Owner instruction 的逐字句子裡。
- 觸發條件：任何寫入/發送/呼叫/刪除動作前。
- 操作：執行 01_SAFETY_BOUNDARIES.md 第 5 節四問。
- PASS：能引用授權句且範圍相符。
- HOLD：引用不出、範圍不符、或授權來自 dashboard/Hermes/上一輪/計劃表。
- 正例：「授權引用：instruction 第 12 行『允許寫入 data/audit_dev.jsonl』→ 執行。」
- 反例：「計劃表 Phase 7 說要寫 audit 檔，所以我寫了。」（計劃≠授權）

## R-07 何時應該拒絕或降級成 safe alternative

- 觸發條件：指令要求違反 01 的動作，或要求「快速跳過驗收」。
- 操作：不執行原樣要求；提出 safe alternative（dry-run 版、preview 版、docs-only 版）+ 說明差距；由 Owner 決定。
- PASS：alternative 保留了原需求的可驗證價值。
- HOLD：Owner 逐字堅持且動作在 01 第 2 節內 → 提醒一次 01 條文；若 Owner 以正式 instruction 逐字授權（01 §4 的 per-phase lift），即可執行，**不需要**修改 01；只有要長期常態開放時才走 40 F2 修改 01。
- 正例：「要求直接呼叫 real OpenClaw → 改為：mock gateway 全流程 + real call 的 command envelope preview 供審。」
- 反例：因為 Owner 語氣急就直接跳過 gate。

## R-08 何時能批次執行 / 何時不能

- 能批次（全部滿足）：模式已由強模型驗證過一個樣本、每項獨立可回滾、失敗可逐項識別、不含 01 禁止動作。
- 不能批次：首次執行的動作類型、涉及寫入共享資源、每項需要語境判斷。
- 操作：批次前先做 1 個樣本 → 驗收 → 才放大；批次中每 N 項抽查一項。
- PASS：樣本驗收紀錄存在。
- HOLD：批次中發現 2 個異常 → 停批，回報。
- 正例：一個 schema 定稿後批量產出其餘 8 個。
- 反例：一口氣改 200 個檔案再一起看結果。

## R-09 何時需要 fresh-context review

- 觸發條件（任一）：產物涉及安全邊界/寫入路徑/執行閘；產物將被 Owner 簽核；產物是制度文件；你自己寫的東西自己驗收超過兩輪。
- 操作：按 30_DELEGATION_PROMPTS.md T-06/T-07 派 fresh subagent。
- PASS：review 回報落檔或進回覆。
- HOLD：無 subagent → 標註自驗（見 C6）。
- 正例：approval packet builder 完成 → adversarial review 找「packet 到執行的通路」。
- 反例：對 typo 修正也開三輪 review（過度使用，浪費）。

## R-10 何時需要 web verification

- 觸發條件：回答將包含——外部工具/repo/API 的存在或行為、模型名稱與能力、價格、日期敏感資訊、「最佳實務」宣稱。
- 操作：先查（WebSearch/WebFetch/實跑），引用來源；查不到寫「無法驗證」。
- PASS：每個外部專有名詞主張都可對應到本 session 的一次工具呼叫。
- HOLD：無網路工具 → 明寫「本環境無法查證，以下憑訓練記憶，可能過時」。
- 正例：「HumanLayer SDK 已 deprecated（來源：本 session 抓取其 README）。」
- 反例：「大家都用 X 框架做這個」（無來源）。

## R-11 何時需要三源一致性驗證

- 觸發條件：session 開工時；任何 push/deploy 決策前；引用「線上狀態」前。
- 操作：跑 00_QUICK_DIAGNOSIS.md D-12 的命令。
- PASS：三 hash/狀態一致。HOLD：不一致 → 報差異等 Owner。
- 正例：開工先跑 D-12 命令，三個 hash 一致才動工。
- 反例：沿用上個 session 記憶中的 HEAD hash 當作現況（細節見 00 D-12）。

## R-12 何時需要 Owner explicit authorization

- 判準：見 01 第 2 節清單 + 40 F2 清單。速記：**會離開本機、會持久化、會被外部看到、會刪東西、會改安全規則**的，都要。
- 操作：R-06 四問。
- PASS：四問全數通過（能引用授權句，或屬 01 §4 第 5 條的任務授權範圍）。
- HOLD：四問任一答不出 → HOLD（fail closed）。
- 正例：寫 fixtures/local_mock_data/ 下 synthetic fixture，**且該寫入是本輪已授權 phase 任務的必要部分** = 屬任務授權範圍（01 §4 第 5 條），不需逐檔授權句，但必須在回報中列出。與本輪任務無關的寫入，仍需授權。
- 反例：「只是寫個測試用的 webhook receiver」——receiver 是新 endpoint，要授權。

## R-13 高風險審查的多審查員規則（二次補強，Owner 裁決 2026-07-08；範例 2026-07-18 修訂）

- 判準：高風險產物的審查絕不交給單一 agent 或單一模型（Owner 原話裁決，見 05 §6.0 Q17）。
- 觸發條件：R-09 的任一觸發條件成立，且產物屬——安全邊界修訂、寫入路徑代碼、執行閘代碼、將供 Owner 簽核的產物。
- 操作：派 **≥2 個不同模型**的 fresh-context 審查員（例：一個 sonnet + 一個 opus；或本環境模型 + Codex Sol+max 對抗審查——跨模型保險，見 10 C8。原「GPT 審查包經 Owner 搬運」路徑已隨 ChatGPT 審查線退場廢止，2026-07-18 修訂）。兩份回報分歧時：分歧點逐條列出交 Owner，不得自行仲裁後沉默。
- PASS：兩份獨立審查回報都落檔，分歧已解決或已交 Owner。
- HOLD：本環境只有單一模型可用 → 標註「單模型審查，不足以作為簽核依據」，等 Owner 決定（改派 Codex 對抗審查或接受降級）。
- 正例：「audit writer 代碼：sonnet fresh review + opus fresh review 各一份，1 個分歧（檔案鎖）已列給 Owner。」
- 反例：「我自己又檢查了一遍，沒問題。」（單一模型 + 自驗，雙重違規）
