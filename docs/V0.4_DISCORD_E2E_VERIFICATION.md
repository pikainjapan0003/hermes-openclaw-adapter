# v0.4 Discord End-to-End Verification

**Date:** 2026-06-14
**Tag:** `v0.4.1-discord-e2e-verified`
**Scope:** Verify the full live chain **Discord → Hermes → MCP → Adapter v0.4 → OpenClaw → `results.jsonl`** after switching the production Adapter on port 8000 to v0.4.
**Out of scope:** v0.5 / Queue / DLQ (intentionally NOT done). No changes to Adapter v0.4 code, Hermes core, or OpenClaw core.

---

## Summary

The Discord bot could not previously reach OpenClaw: a Discord task only triggered `skill_view` and the model answered `PONG` itself, never calling `dispatch_to_openclaw` (no `POST /tasks/dispatch` reached the Adapter). Root cause was a **stale long-running Hermes Gateway** that had loaded its MCP tool registry at startup (2026-06-11), *before* the `openclaw` MCP server was registered in config (2026-06-14). Restarting the Gateway made it reload the registry; the full chain then verified green.

## 2. Gateway restart — reason

The Discord bot is **not** a separate process; it runs **inside the Hermes Gateway** (`hermes gateway run`, detached tmux session `hermes`, user `lnovo`, `HOME=/home/lnovo`, default `~/.hermes/config.yaml`, no `HERMES_PROFILE` override → the **same** profile used by `hermes -z` CLI tests).

Timeline that pinned the root cause:

| Event | Time |
|---|---|
| Gateway (old, pid 582) started | 2026-06-11 06:00:57 |
| `openclaw` MCP written into `config.yaml` | 2026-06-14 15:55:43 |

The long-running gateway loaded its MCP tool registry **once at startup** (6/11), before the MCP existed (6/14) → Discord sessions never saw the tools. Fresh `hermes -z` sessions did, because they start cold and read the current config.

Gentle restart performed:

```
tmux send-keys -t hermes C-c          # graceful stop (logs "Gateway stopped (teardown 0.63s)")
tmux new -d -s hermes hermes gateway run   # relaunch in fresh tmux session
```

- Old gateway shut down cleanly at 19:05:03 (even sent a Discord shutdown notification).
- New gateway: pid 19282, started 19:06:37.

## 3. Discord stale session deletion

The Discord channel (`1512822248342556702`) had been continuing session `20260614_165027_152f0d89` (the 16:50 "Discord PONG 安全測試" session), reused at 18:54 with `history=14` — a history short-circuit risk. Deleted:

```
$ printf 'y\n' | hermes sessions delete 20260614_165027_152f0d89
Deleted session '20260614_165027_152f0d89'.
```

Note: `hermes sessions delete` has an interactive `[y/N]` prompt — pipe `y` in non-TTY. Do **not** `pkill -f 'sessions delete'` (it self-matches your own command line).

## 8. Root cause

A long-running Hermes Gateway loads its MCP tool registry **once, at startup**. The `openclaw` MCP server was registered into `~/.hermes/config.yaml` *after* the gateway was already running, so the live gateway (and therefore every Discord session it served) had a stale tool list with no `dispatch_to_openclaw`. CLI `hermes -z` worked only because each invocation starts cold and reads the current config. The Adapter, MCP registration, and profile were all already correct — the only required fix was a gateway restart.

## 9. Operational rule — adding MCP tools later

> **Whenever you register/change an MCP server (`hermes mcp add/remove/configure`), you MUST restart the Hermes Gateway** for Discord / messaging sessions to pick it up. Verify reload via `agent.log` line `MCP server '<name>' (stdio): registered N tool(s): …` and `[Discord] Connected`.

Restart recipe:

```
tmux send-keys -t hermes C-c          # graceful stop
tmux new -d -s hermes hermes gateway run
```

## 10. Current limitations

- **Neither the Adapter (port 8000) nor the Hermes Gateway is started on boot.** After a WSL/machine restart both must be relaunched manually. A future hardening step (out of scope here) is a systemd unit for each.
- Adapter still uses BackgroundTasks (no Queue / DLQ / native webhook) — that is the deferred v0.5 work.
